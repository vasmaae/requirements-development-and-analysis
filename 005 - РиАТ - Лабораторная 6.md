### 1. Общие сведения

Система управления аптекой и складами (СУАС) предназначена для автоматизации процессов учета лекарственных средств (ЛС), медицинских изделий и расходных материалов в больнице. АС является самостоятельной системой, интегрируемой с больничной информационной системой (электронные карты пациентов) и внешними API (Минздрав РФ для МНН).  

Перечень документов, на основании которых создается АС:  

- Спецификация требований к ПО (версия 1.3 от 30.10.2025, утверждена заказчиком 30.10.2025).  
- Федеральный закон РФ № 61-ФЗ "Об обращении лекарственных средств" (утвержден 12.04.2010).  
- Федеральный закон РФ № 152-ФЗ "О персональных данных" (утвержден 27.07.2006).  
- Стандарт IEEE 830-1998 "Recommended Practice for Software Requirements Specifications".  
- ГОСТ 19.201-78 "Единая система программной документации. Техническое задание".  
- Национальный проект "Здравоохранение" (Минздрав РФ).  

### 2. Цели и назначение создания автоматизированной системы

#### 2.1. Цели создания АС

Цели создания АС:  

- Автоматизация учета ЛС, медицинских изделий и материалов для минимизации ошибок ручного учета (целевое значение: снижение ошибок на 90%, критерий оценки - сравнение с текущими ручными процессами по журналам и Excel).  
- Обеспечение контроля остатков и сроков годности для предотвращения потерь от просрочки (целевое значение: снижение потерь на 80%, критерий - отчеты по списаниям).  
- Улучшение эффективности логистики: формирование заявок, резервирование и списание ЛС (целевое значение: сокращение времени на обработку заявок с 1 часа до 5 минут, критерий - измерение времени операций).  
- Соответствие регуляторным требованиям (ФЗ-61, ФЗ-152) и цифровизации здравоохранения (целевое значение: 100% соответствие, критерий - аудит Росздравнадзора).  
- Повышение удобства для пользователей за счет когнитивного интерфейса (целевое значение: время обучения <1 часа, критерий - опросы пользователей).  

#### 2.2. Назначение АС

АС предназначена для автоматизации управления аптекой и складами в больнице, включая учет, контроль, отчетность и взаимодействие с отделениями. Вид деятельности: управление логистикой и ресурсами в здравоохранении. АС применяется в рамках одного больничного комплекса, охватывая аптеку и склады.

### 3. Характеристика объектов автоматизации

Объектом автоматизации является аптека и склады больницы, включая процессы учета ЛС по МНН и торговым названиям, медицинских изделий, контроля остатков и сроков годности, формирования заявок от отделений, резервирования и списания.  

Основные сведения об объекте:  

- Больничный комплекс с до 50 пользователями (кладовщики, врачи, администраторы).  
- Текущие процессы: ручные журналы и Excel-таблицы, подлежащие замене.  
- Условия эксплуатации: локальная сеть больницы, стабильный интернет для интеграций; окружающая среда - стандартные условия медицинского учреждения (температура 18-25°C, влажность 40-60%, без экстремальных воздействий).  

Ссылка на документы: Спецификация требований (раздел 2.1 Общий взгляд на продукт).

### 4. Требования к автоматизированной системе

#### 4.1. Требования к структуре АС в целом

АС состоит из подсистем:  

- Подсистема учета ЛС и материалов (назначение: CRUD-операции, поиск, импорт; характеристики: интеграция с БД PostgreSQL).  
- Подсистема контроля (назначение: мониторинг остатков и сроков; характеристики: автоматические уведомления).  
- Подсистема заявок и резервирования (назначение: обработка заявок, резервирование; характеристики: интеграция с ЭМК).  
- Подсистема отчетности (назначение: генерация отчетов; характеристики: экспорт в PDF/Excel).  

Требования к взаимодействию: информационный обмен через REST API (JSON), совместимость со смежными системами (ЭМК, API Минздрава). Режимы функционирования: непрерывный (24/7) с диагностикой сбоев. Перспективы: масштабирование до 100 пользователей, добавление мобильного приложения.  

Ссылка на ГОСТ 19.201-78.

#### 4.2. Требования к функциям (задачам), выполняемым АС

Перечень функций:  

- Учет ЛС по МНН (результат: обновленная БД, точность 100%, время <1 сек).  
- Учет медицинских изделий (аналогично).  
- Контроль сроков годности (результат: уведомления и автосписание, ежедневно).  
- Учет остатков (результат: дашборд в реальном времени).  
- Формирование заявок (результат: обработанная заявка, время <2 мин).  
- Резервирование ЛС (результат: заблокированные остатки).  
- Списание ЛС (результат: обновленные остатки).  
- Генерация отчетов (результат: PDF/Excel файл, время <5 сек).  

Критерии отказов: MTBF >99.9%, для критических функций (учет, списание).  

#### 4.3. Требования к видам обеспечения АС

- Математическое: использование алгоритмов валидации (например, проверка сроков), без сложных моделей.  
- Информационное: структура данных в PostgreSQL (таблицы для ЛС, остатков), обмен JSON, совместимость с классификаторами МНН.  
- Лингвистическое: русский язык интерфейса, поддержка автозаполнения.  
- Программное: Golang + Vue 3, интеграция с PostgreSQL, без покупки ПО.  
- Техническое: серверы (Intel i5, 8 ГБ RAM), ОС Windows 10+/Linux, сканеры штрих-кодов.  
- Метрологическое: точность учета (количество ±1 единица), методы контроля - логи.  
- Организационное: структура - IT-отдел, порядок взаимодействия по ролям.  
- Методическое: соответствие ФЗ-61, ФЗ-152, IEEE 830.  

#### 4.4. Общие технические требования к АС

- Численность персонала: 1-5 кладовщиков, обучение <1 час.  
- Показатели назначения: отклик <2 сек, пропускная способность 200 операций/мин.  
- Надежность: MTBF >99.9%, автобэкап ежедневно.  
- Безопасность: шифрование AES-256, ролевой доступ.  
- Эргономика: Material Design, экранные ридеры.  
- Защита информации: от НСД по ФЗ-152.  
- Стандартизация: использование типовых решений (PostgreSQL, REST).
